# Requirements: duplicate-emhass-sensor-fix

## Goal

Fix two bugs in the EV Trip Planner Home Assistant integration: (1) orphaned EMHASS state-based sensors surviving after integration deletion, and (2) sensor attributes not updating when trips or SOC change.

## User Stories

### US-1: Remove Orphaned EMHASS Sensors on Integration Unload

**As a** user
**I want** old EMHASS sensors to be completely removed when I delete an integration
**So that** I do not see duplicate/orphaned sensors when I create a new integration with a different name

**Acceptance Criteria:**
- [ ] AC-1.1: When `async_unload_entry()` is called, all state-based EMHASS sensors created by `publish_deferrable_loads()` are removed via `hass.states.async_remove()`
- [ ] AC-1.2: `async_cleanup_vehicle_indices()` calls `hass.states.async_remove()` for each entity, not `hass.states.async_set(state, "idle")`
- [ ] AC-1.3: `async_unload_entry()` calls `emhass_adapter.async_cleanup_vehicle_indices()` before unloading platforms
- [ ] AC-1.4: State-based entity tracking is maintained so cleanup knows which entities to remove

### US-2: Persist Sensor Attribute Updates

**As a** user
**I want** `power_profile_watts` and `deferrables_schedule` attributes to update when trips or car SOC change
**So that** the sensor always shows current charging schedule information

**Acceptance Criteria:**
- [ ] AC-2.1: `EmhassDeferrableLoadSensor.async_update()` calls `self.async_schedule_update_ha_state()` after updating `_cached_attrs`
- [ ] AC-2.2: After a trip is saved or SOC changes, the sensor's `power_profile_watts` reflects the new calculated values
- [ ] AC-2.3: After a trip is saved or SOC changes, the sensor's `deferrables_schedule` reflects the new schedule

### US-3: Clean Orphaned Sensors at Startup (SAFE)

**As a** user
**I want** orphaned sensors from previously deleted integrations to be cleaned at startup
**So that** stale sensors do not clutter my Home Assistant instance

**Acceptance Criteria:**
- [ ] AC-3.1: At startup, the integration scans for state-based EMHASS sensors with `entry_id` attributes
- [ ] AC-3.2: If a sensor's `entry_id` does not exist in `hass.config_entries.async_entries()`, it is removed
- [ ] AC-3.3: Sensors belonging to OTHER active vehicles are NOT removed (SAFE - only removes truly orphaned sensors)
- [ ] AC-3.4: If a sensor's `entry_id` cannot be confidently determined to be orphaned, it is NOT removed

## Functional Requirements

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-1 | Orphaned EMHASS state-based entities are removed when integration is unloaded | High | `async_unload_entry()` triggers removal of all entities created by `publish_deferrable_loads()` via `hass.states.async_remove()` |
| FR-2 | `async_cleanup_vehicle_indices()` calls `async_remove()` not `async_set(idle)` | High | Each entity tracked by the adapter is removed using `hass.states.async_remove(entity_id)`, not set to "idle" state |
| FR-3 | `async_update()` calls `async_schedule_update_ha_state()` to persist attribute changes | High | After updating `_cached_attrs` with new `power_profile_watts` and `deferrables_schedule`, `self.async_schedule_update_ha_state()` is called |
| FR-4 | At startup, orphaned sensors from previously deleted integrations are cleaned | Medium | During `async_setup_entry()`, scan for sensors with stale `entry_id` and remove only those whose `entry_id` is not in `hass.config_entries.async_entries()` |
| FR-5 | Cleanup at startup is SAFE - only removes sensors whose `entry_id` is not in active config entries | High | If uncertain whether a sensor belongs to an active vehicle, do NOT delete it |

## Glossary

- **Orphaned sensor**: A state-based entity created by `hass.states.async_set()` whose integration entry no longer exists in `hass.config_entries`. The entity persists in HA's state machine but has no associated config entry.
- **State-based entity**: An entity created directly in HA's state machine via `hass.states.async_set()`, bypassing the entity registry. Not automatically cleaned up when the integration unloads.
- **Config entry ID**: A unique identifier (`entry_id`) generated by Home Assistant for each integration instance. Used to tag sensors and determine their association.
- **Entity registry**: HA's internal registry that tracks entities and handles automatic cleanup. State-based entities created via `hass.states.async_set()` do NOT use this mechanism.

## Out of Scope

- Changes to entity registry-based `EmhassDeferrableLoadSensor` creation/cleanup (already works correctly via `async_add_entities()`)
- Changes to trip CRUD functionality
- Changes to SOC calculation logic
- Changes to dashboard panel or UI components
- Migration of existing state-based entities to entity registry-based entities

## Dependencies

- Home Assistant async state machine APIs: `hass.states.async_set()`, `hass.states.async_remove()`
- `hass.config_entries.async_entries()` to enumerate active config entries
- `hass.config_entries.async_unload_platforms()` (existing unload mechanism)
- Entity update scheduling: `self.async_schedule_update_ha_state()`

## Success Criteria

1. Delete integration "chispitas" and create "chispis" - only one EMHASS sensor exists for "chispis"
2. Save a trip with low SOC - sensor `power_profile_watts` and `deferrables_schedule` update within 5 seconds
3. Restart Home Assistant after deleting an integration - no orphaned sensors appear in the new integration
4. Multiple vehicles configured - deleting one vehicle does NOT affect the other's sensors

## Verification Contract

**Project type**: `api-only` (Home Assistant integration with no browser UI)

**Entry points**:
- `async_setup_entry()` - integration setup
- `async_unload_entry()` - integration unload (triggers cleanup)
- `EmhassDeferrableLoadSensor.async_update()` - entity update cycle
- `EmhassDeferrableLoadSensor.async_will_remove_from_hass()` - entity removal
- `emhass_adapter.publish_deferrable_loads()` - state-based entity creation

**Observable signals**:
- PASS: After unloading entry, `hass.states.async_remove()` called for each state-based entity
- PASS: After `async_update()`, `async_schedule_update_ha_state()` called and attributes reflect new values
- PASS: After startup with orphaned sensors, those with stale `entry_id` are removed
- FAIL: Duplicate sensor IDs in `hass.states.async_all()` after integration recreation
- FAIL: `power_profile_watts` remains stale after trip save (not updated within 5s)

**Hard invariants**:
- Auth/session validity: N/A (no auth in this fix)
- Permissions: Only affect the integration's own entities
- Data belonging to other users/tenants: N/A (single-user HA instance)
- Adjacent unrelated flows: Trip CRUD, SOC calculations, dashboard panel must continue working

**Seed data**:
- At least one EMHASS adapter instance with published sensors
- For startup cleanup testing: a previously deleted integration's sensor that was not cleaned up

**Dependency map**:
- `emhass_adapter.py` - owns state-based entity creation and tracking
- `sensor.py` - `EmhassDeferrableLoadSensor` entity class
- `__init__.py` - integration entry point and unload logic
- `trip_manager.py` - calls `publish_deferrable_loads()` when trips change

**Escalate if**:
- `hass.states.async_remove()` behavior differs from expected (may not fire `async_will_remove_from_hass()`)
- Entity registry entities and state-based entities share same sensor ID namespace causing conflicts
- Startup cleanup finds sensors that appear to belong to existing entries but are actually stale
