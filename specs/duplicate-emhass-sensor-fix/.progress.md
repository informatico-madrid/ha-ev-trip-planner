# Progress: duplicate-emhass-sensor-fix

## Original Goal

Corrección de dos bugs relacionados con sensores EMHASS:

1. **Sensor duplicado**: Al crear un vehículo (integración) y luego crear un viaje desde el panel de control, se crean dos sensores EMHASS casi idénticos (uno sobra).

2. **Atributos del sensor no se actualizan**: Cuando se modifican viajes o el SOC del coche, `power_profile_watts` y `deferrables_schedule` no se actualizan aunque `last_update` sí cambia.

## Learnings

### Bug 1: Duplicate Sensor After Integration Re-creation

- **Root Cause**: `emhass_adapter.publish_deferrable_loads()` crea state-based entities via `hass.states.async_set()` que NO se limpian con `async_unload_entry()`
- **Two mechanisms**: Entity registry (`EmhassDeferrableLoadSensor`) vs state machine (`hass.states.async_set()`)
- **Why chispitas survives**: `async_unload_entry()` no llama a `async_cleanup_vehicle_indices()`. El sensor queda huérfano en HA state machine
- **Cleanup is wrong**: `async_cleanup_vehicle_indices()` pone estado "idle" pero no llama `async_remove()`
- **entry_id**: Es el HA config entry_id, no vehicle_id. Cada re-creación de integración genera nuevo entry_id
- **Key files**: `emhass_adapter.py:509-520`, `emhass_adapter.py:1105-1163`, `__init__.py:775-819`, `sensor.py:624-627`

### Bug 2: Attributes Not Updating

- **Root Cause**: `EmhassDeferrableLoadSensor.async_update()` set `_cached_attrs` pero NO llama `async_schedule_update_ha_state()` para persistir
- **Two conflicting paths**:
  - Path 1 (trip changes): `async_save_trips()` → `publish_deferrable_loads()` → `hass.states.async_set()` directa
  - Path 2 (entity update): `async_update()` → genera nuevos valores → actualiza `_cached_attrs` → (FALTA `async_schedule_update_ha_state()`)
- **Why last_update works**: `publish_deferrable_loads()` NO incluye `last_update` en su `hass.states.async_set()`. Cada vez que `async_update()` corre, actualiza `self._cached_attrs["last_update"]` con timestamp fresco
- **But others don't**: `power_profile_watts` y `deferrables_schedule` fueron establecidos por Path 1 y los nuevos valores de Path 2 nunca llegan al state machine de HA
- **Fix**: Añadir `self.async_schedule_update_ha_state()` al final del try block en `async_update()` (sensor.py:~605)

## Design Phase Learnings

- **Test runner confirmed**: pytest is the test runner (`pytest-homeassistant-custom-component` plugin), NOT jest. Jest config expects `.test.js` but tests are Python-based
- **Pytest collection verified**: Tests collected successfully with `pytest tests/test_sensor.py -v --co`
- **Entity tracking approach**: Adding `_published_entity_ids: Set[str]` to EMHASSAdapter is minimal intrusion - only 1 new field
- **Cleanup timing matters**: Must call `async_cleanup_vehicle_indices()` BEFORE `async_unload_platforms()` so entities still exist when we try to remove them
- **async_remove is idempotent**: Safe to call even if entity doesn't exist - no error thrown
- **Startup cleanup uses entry_id attribute**: The orphaned sensor stores `entry_id` in its attributes (set by `publish_deferrable_loads()`) - can use this to match against `hass.config_entries.async_entries()`
- **async_schedule_update_ha_state()**: This schedules an async state update, doesn't execute synchronously - safe to call in tests as mock
