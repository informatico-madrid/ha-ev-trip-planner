## Task Planning Learnings

- **TripPlannerSensor base class** (sensor.py:64) ŌĆö currently `SensorEntity` only, NO `_attr_unique_id`, NO `CoordinatorEntity`. 7 subclasses inherit from it without setting unique_id.
- **async_create_trip_sensor** (sensor.py:1019) ŌĆö creates orphan Python objects stored in `hass.data[DATA_RUNTIME][namespace]["trip_sensors"]`, NEVER calls `async_add_entities()`. The correct path is `_async_create_trip_sensors` (sensor.py:939) which IS passed to `async_add_entities()`.
- **async_remove_trip_sensor** (sensor.py:1132) ŌĆö only deletes from dict (`del trip_sensors[trip_id]`), never calls `entity_registry.async_remove()`. Zombies the registry.
- **EmhassDeferrableLoadSensor** (sensor.py:494) ŌĆö already HAS `_attr_unique_id` and calls `async_schedule_update_ha_state()`, but does NOT inherit from `CoordinatorEntity`. Two writers conflict: `publish_deferrable_loads()` via `hass.states.async_set()` and `EmhassDeferrableLoadSensor.async_update()`.
- **publish_deferrable_loads** (emhass_adapter.py:534) ŌĆö uses `hass.states.async_set()` which bypasses entity registry, creates state-only entities that conflict with proper `EmhassDeferrableLoadSensor` entities.
- **sensor_async_add_entities callback** ŌĆö captured from platform setup (sensor.py async_setup_entry) into EVTripRuntimeData.sensor_async_add_entities. Service handlers use this for dynamic TripSensor creation.
- **coordinator.data full contract** ŌĆö EMHASS keys (emhass_power_profile, emhass_deferrables_schedule, emhass_status) defined as None in Phase 1 task 1.2, populated in Phase 3 task 3.4.
- **Phase 0 tests are characterization tests** ŌĆö They document broken behavior (FAIL today) and verify correct behavior after fix (PASS after). Not testing code correctness, testing that the refactor achieves its goal.

- **Bug TDD workflow**: Phase 0 (reproduce) + Phase 1-4 (TDD) + Phase 5 (cleanup) + VF (verification final)
- **E2E tests untouched** ŌĆö 16 existing E2E tests remain as safety net per user requirement
