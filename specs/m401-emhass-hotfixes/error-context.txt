## Root Cause Analysis

### Data Flow:
1. Trip created via `panel.js:trip_create` service call
2. `services.py:handle_trip_create()` → `trip_manager.async_add_punctual_trip()`
3. `trip_manager._async_publish_new_trip_to_emhass()` → `emhass_adapter.async_publish_deferrable_load()`
4. `emhass_adapter.async_publish_all_deferrable_loads()` calculates power profile
5. `emhass_adapter._calculate_power_profile_from_trips()` → `calculations.calculate_power_profile_from_trips()`
6. Result stored in `emhass_adapter._cached_power_profile`
7. `coordinator._async_update_data()` reads via `emhass_adapter.get_cached_optimization_results()`
8. Sensor reads from `coordinator.data.get("emhass_power_profile")`

### Issue:
The sensor exists with `emhass_status: ready` but `power_profile_watts: [0,0,0,0,0]`.

This means:
- EMHASS adapter is initialized and running
- The sensor entity is created
- But the power profile calculation returns zeros

### Hypothesis:
Looking at `emhass_adapter.py:335-338`:
```python
hours_available = (deadline_dt - now).total_seconds() / 3600

if hours_available <= 0:
    _LOGGER.warning("Trip deadline in past: %s", trip_id)
    await self.async_release_trip_index(trip_id)
    return False  # Trip is rejected
```

If trips are rejected here due to deadline being calculated as "in the past", they never reach the power profile calculation.

### Potential Causes:
1. **Timezone mismatch**: Test runs in UTC, but `datetime.now()` might use local timezone
2. **Datetime format issue**: The `datetime` field from HTML5 datetime-local input is `YYYY-MM-DDTHH:MM` without seconds
3. **Timing issue**: The trip's "tomorrow" deadline is calculated relative to test start time, but power profile is computed at test read time (5+ seconds later)
