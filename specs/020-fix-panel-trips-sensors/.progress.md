## Requirements Insight from Analysis
- Vehicle ID extraction requires 3 fallback methods (split, regex, hash) due to varying HA URL patterns
- Trip list service response handling must account for 3 formats: direct result, array `[result]`, or object `{result: ...}`
- Sensor value formatting rules are critical: percentages (1 decimal), energy (2 decimals), distance (1 decimal)
- `_rendered` flag must only be set after trips section fully loads to prevent early exit from connectedCallback
- Polling for hass must stop immediately when panel begins rendering to avoid race conditions
- `window._tripPanel` reference must be set before any onclick handlers can work
- Sensor filtering logic changed: unavailable/unknown states show "N/A" instead of being hidden
- State subscription requires lowercase entity ID matching for real-time updates

## Design Learnings
- 3 fallback URL extraction methods (split, regex, hash) ensure robustness across HA versions
- Trip response normalization handles direct, array `[result]`, and object `{result: ...}` formats
- Sensor "N/A" display provides visibility into missing data rather than hiding it
- Partial DOM updates for state changes better than full re-render for performance
- `_rendered` flag timing is critical: must wait for trips section to load to prevent early exit
- Multiple entity patterns needed: `sensor.{vehicleId}`, `sensor.{vehicleId}_`, `sensor.ev_trip_planner_{vehicleId}`, etc.
- Technical design explicitly documents all decisions with options considered and rationale
