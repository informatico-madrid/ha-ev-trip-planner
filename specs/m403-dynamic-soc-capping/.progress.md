# Progress: m403-dynamic-soc-capping

## Goal
Consolidate dynamic SOC capping research, integrate into existing algorithm without breaking what works well.

## Context
- Formula: risk = t * (soc_post_trip - 35) / 65, SOC_lim = 35 + 65 * [1 / (1 + risk/T)]
- Negative risk (soc_post_trip <= 35%) -> 100% allowed
- T_base = 24h default (user-configurable 6-48h)
- Integration: min(required_soc, dynamic_limit) -- trip always wins
- Health mode always-on, no toggle
- SOH sensor needed for real battery capacity

## Learnings
- Config flow sensors step (Step 2) uses STEP_SENSORS_SCHEMA at line 61 — SOH sensor selector goes there
- Config flow options flow (OptionsFlowHandler.async_step_init) only exposes battery, charging, consumption, safety_margin — sensors and battery health NOT editable after initial setup (gap #4 from prior session)
- `calculate_deficit_propagation()` at line 716 iterates in reverse order; integration point is line ~808 after `soc_objetivo_ajustado` computation, within the result-building loop (line ~828+)
- Deficit propagation result loop (lines 828-861) recomputes `soc_objetivo_ajustado` — capping must be applied in both the backward propagation loop AND the result-building loop for consistency
- T_base config should go into a new step (battery health) after sensors step, or merged into options flow since there's no toggle (health mode always on)
- SOH capacity calculation: `capacidad_real = nominal * SOH/100` — this value replaces `battery_capacity_kwh` everywhere (trip manager, calculations, EMHASS adapter)
- `determine_charging_need()` is called from `_populate_per_trip_cache_entry()` in emhass_adapter.py — dynamic limit affects the decision because `soc_objetivo` feeds into energy calculations
- Critical: capping must be applied in `calculate_deficit_propagation()` BEFORE deficit is calculated — otherwise capped SOC may cause artificial deficits to propagate backward
- The formula uses `soc_base = 35` internally hardcoded — not user-configurable despite spec mentioning it as a parameter. User-facing config shows only T_base.
- Requirement insight from analysis: capping in result-building loop (line ~843) must apply min() to produce final soc_objetivo that feeds kwh_necesarios — the backward propagation loop applies the cap for computing accumulated deficits, but the result loop must also apply it for the final output
- Requirements analysis insight: SOH sensor is optional at config time — system must gracefully fall back to nominal capacity if SOH sensor is not configured (no crash, no config error)
- Requirements analysis insight: T_base default of 24h aligns with the industry-validated "24h return buffer" pattern — a car idle for 24h at moderate SOC should get tighter capping than a car returning in 2h

## Learnings

- BatteryCapacity is a new abstraction (no existing equivalent) — designed as a frozen dataclass wrapping nominal capacity + SOH sensor lookup
- SOH sensor read timing is critical: pre-resolving at construction keeps calculations.py pure (no HA I/O in calculation path)
- Deficit propagation uses REVERSE iteration (last trip → first trip) — capping must be applied in BOTH the backward loop AND the result-building loop for consistency
- The forward-propagated SOC for trip N+1 must use the CAPPED soc_objetivo_final (not uncapped soc_objetivo_ajustado) to maintain consistency
- T_base and SOH config must go in BOTH initial sensors step AND options flow — existing pattern in codebase (battery_capacity is in both)
- EMHASS adapter `_populate_per_trip_cache_entry()` already uses battery_capacity_kwh parameter — only change is that parameter becomes real_capacity

- The formula uses soc_base=35 hardcoded internally — this is a chemistry-specific constant (NMC/NCA sweet spot) not a user decision
- SOH sensor fallback to nominal capacity is essential: graceful degradation prevents system failure when sensor is unavailable
