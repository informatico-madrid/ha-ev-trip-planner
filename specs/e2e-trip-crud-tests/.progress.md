### Key Findings

**Best Practices:**
- Home Assistant requires UI-based Config Flow installation (not just login)
- Shadow DOM requires web-first locators: getByRole, getByText, getByLabel
- auth.setup.ts pattern: login + Config Flow + storageState save
- Anti-patterns: waitForTimeout, CSS/XPath selectors, hardcoded URLs

**CRUD Operations to Test:**
- Create: trip_create service via "+ Agregar Viaje" button
- Read: trip_list service displaying trips
- Update: trip_update via "Editar" button
- Delete: delete_trip via "Eliminar" + confirmation
- Pause/Resume: pause_recurring_trip/resume_recurring_trip services
- Complete/Cancel: complete_punctual_trip/cancel_punctual_trip services

## Design Learnings

- Panel URL pattern is `http://127.0.0.1:8477/ev-trip-planner-Coche2` (stored in `playwright/.auth/panel-url.txt`)
- EVTripPlannerPage at `tests/e2e/pages/ev-trip-planner.page.ts` exists but focuses on dashboard/vehicle cards, not trips specifically - TripsPage should be new
- test-helpers.ts contains TripPanel class with Shadow DOM traversal via `page.locator('ev-trip-planner-panel').locator('.trip-card')` - reference for locator strategy
- auth.setup.ts uses `waitForTimeout` in several places (lines 109, 148, 191, 217, 227, 235) due to CI rendering issues - but tests should NOT use waitForTimeout per requirements
- Playwright config at `playwright.config.ts` has `storageState: 'playwright/.auth/user.json'` injected into chromium project automatically
- TripPanel class in test-helpers.ts has useful patterns: `setupDialogHandler`, `waitForFormToClose`, `fillTripForm` - but uses CSS class locators which should be avoided in favor of web-first
- Service names confirmed: trip_create, trip_update, delete_trip, pause_recurring_trip, resume_recurring_trip, complete_punctual_trip, cancel_punctual_trip
- UI button labels confirmed: "+ Agregar Viaje", "Editar", "Eliminar", "Pausar", "Reanudar", "Completar", "Cancelar"
