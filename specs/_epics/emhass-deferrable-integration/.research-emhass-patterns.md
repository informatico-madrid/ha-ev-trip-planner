# Research: EMHASS Integration Patterns

## Executive Summary

EMHASS (Energy Management Home Assistant) is an energy optimization addon that uses MPC (Model Predictive Control) to schedule deferrable loads. The EV Trip Planner integrates by publishing trip energy requirements as deferrable loads and reading EMHASS's optimized schedules to control charging. This research covers the data formats, API expectations, and patterns for successful integration.

---

## 1. EMHASS Deferrable Load Concept

### 1.1 What is a Deferrable Load?

A deferrable load is an appliance that:
- Has a **nominal power** (P_deferrable_nom)
- Has a **total energy requirement** (def_total_hours * P_nominal)
- Can be scheduled within a **time window** (def_start_timestep to def_end_timestep)
- May have constraints like being semi-continuous (can pause/resume)

Example: EV charging
- Nominal power: 3600W (3.6kW)
- Total energy: 30kWh
- Time window: 24 hours (0 to 24 timesteps, each timestep = 1 hour)
- Result: needs ~8.3 hours to complete

### 1.2 EMHASS Planning Cycle

1. **User defines deferrable loads** via sensors or configuration
2. **EMHASS reads current state** - SOC, solar production, grid prices
3. **EMHASS runs optimization** - MPC solves for optimal schedule
4. **EMHASS writes output** - schedule to sensors (p_deferrable0, p_deferrable1, etc.)
5. **Home automation reads output** - controls devices based on schedule
6. **Cycle repeats** - typically every 5-15 minutes

## 2. EMHASS Data Format

### 2.1 Input: Deferrable Load Configuration

**Sensor**: `sensor.emhass_deferrable_load_config_{index}`

**State**: "active" | "idle"

**Attributes**:
```python
{
    "def_total_hours": [5.56],      # Hours needed at nominal power
    "P_deferrable_nom": [3600],     # Power in Watts
    "def_start_timestep": [0],      # 0 = ASAP, else specific timestep
    "def_end_timestep": [24],       # Deadline hour index
    "treat_deferrable_load_as_semi_cont": [true],  # Can pause?
    "set_deferrable_load_single_constant": [true],  # Constant power?
    "trip_id": "rec_lun_abc123",
    "vehicle_id": "coche1",
    "kwh_needed": 20.0,
    "deadline": "2026-03-29T22:00:00",
    "emhass_index": 0,
}
```

### 2.2 Input: Power Profile (Aggregated)

**Sensor**: `sensor.emhass_perfil_diferible_{vehicle_id}`

**State**: "ready" | "active" | "error"

**Attributes**:
```python
{
    "power_profile_watts": [0.0, 0.0, 3600.0, 3600.0, ...],  # 168 values (7 days)
    "deferrables_schedule": [
        {"date": "2026-03-29T14:00:00+01:00", "p_deferrable0": "0.0", "p_deferrable1": "3600.0"},
        ...
    ],
    "vehicle_id": "coche1",
    "trips_count": 2,
    "emhass_status": "ok",  # or "error"
    "error_type": "...",    # if error
    "error_message": "...", # if error
}
```

### 2.3 Output: Optimized Schedule (from EMHASS)

EMHASS writes optimized values to sensors like:
- `sensor.emhass_deferrable0` - schedule for load index 0
- `sensor.emhass_deferrable1` - schedule for load index 1
- `sensor.emhass_plan_{vehicle}_mpc_congelado` - full MPC plan

**Schedule format**: Either:
- Time series: `["00:00-01:00", "02:00-05:00"]` 
- Or JSON: `[{"date": "...", "p_deferrable0": "0.0"}, ...]`

### 2.4 EMHASS Command Reference

The shell command (user-configured in configuration.yaml) typically POSTs to:
```
http://localhost:5000/emhass/optimize
```

With payload containing all deferrable load configurations.

## 3. Index Assignment Patterns

### 3.1 Index as Load Identifier

The index (0, 1, 2, ...) is a **stable identifier** for a specific load:
- Index 0 = Trip A (morning commute)
- Index 1 = Trip B (evening commute)

EMHASS optimizes and outputs schedules using these indices.

### 3.2 Index Stability Importance

**Problem**: If Trip A is deleted and indices shift:
- Old: Trip A=0, Trip B=1
- New (after delete): Trip B=0

EMHASS would interpret index 0 as Trip B's load, breaking the schedule.

**Solution (not implemented)**: Maintain stable indices:
- When trip is deleted, don't reassign its index
- When trip is added, use smallest available index
- Allow "holes" in index space

### 3.3 Current Implementation Issues

From `emhass_adapter.py`:

```python
# Index pool is list of available indices
self._available_indices: List[int] = list(range(self.max_deferrable_loads))

# Assign smallest available
assigned_index = min(self._available_indices)

# Release returns index to pool
released_index = self._index_map.pop(trip_id)
self._available_indices.append(released_index)
```

**Issues**:
1. Deleted trip's index is reused by next new trip
2. No guarantee index 0 won't jump to different trip
3. EMHASS may still track old index until it times out

### 3.4 Recommended Index Stability Algorithm

```
When adding trip:
    if trip_id already has index:
        return existing index  # Stable for updates
    elif available_indices not empty:
        assign min(available_indices)
        remove from available
    else:
        return None  # No indices available

When removing trip:
    mark index as "released" but don't reuse immediately
    OR: keep index reserved for a cooldown period

When republishing (after edit):
    reuse same index (already implemented correctly)
```

## 4. Charging Window Calculation

### 4.1 Window Definition

The **charging window** is the time between:
- **Start**: When vehicle returns home AND is plugged in
- **End**: When next trip departs (deadline)

### 4.2 Current vs. Required Calculation

**Current (trip_manager.py)**:
```python
# From async_generate_power_profile()
horas_hasta_viaje = int((trip_time - now).total_seconds() / 3600)
hora_inicio_carga = max(0, horas_hasta_viaje - horas_necesarias)
```

Assumes: charging can start "now" (or at hour 0).

**Required for this epic**:
```python
# Window should start when car returns, not when trip is created
ventana_inicio = max(hora_regreso, ahora)
ventana_fin = hora_salida_proximo_viaje
horas_disponibles = ventana_fin - ventana_inicio
```

### 4.3 Multi-Trip Window Chaining

With consecutive trips:
```
Trip A (morning 08:00) → Car leaves home → Returns 10:00 → Trip B (evening 18:00)
                                           ↑
                                    Charging window A→B
```

Window calculation must account for **return time** from previous trip.

## 5. SOC Milestone Algorithm

### 5.1 SOC Target Calculation

For each trip, SOC target = energy_for_trip + buffer

```
buffer = battery_capacity * safety_margin_percent
# Or: buffer = max fixed value, e.g., 10% of battery
```

### 5.2 Deficit Propagation

**Scenario**:
- Trip morning (08:00): needs 30% SOC + 10% buffer = 40% target
- Trip evening (18:00): needs 80% SOC
- Window between trips: 4 hours at 10% SOC/hour = 40% gained
- Evening arrival SOC: 40% + 40% = 80% → meets target, no deficit

**With deficit**:
- Window only 2 hours at 10% SOC/hour = 20% gained
- Evening arrival SOC: 40% + 20% = 60%
- Deficit: 80% - 60% = 20%
- **Propagate**: Morning target += 20% → 60% total

### 5.3 Algorithm Pseudocode

```python
def calcular_hitos_soc(trips_sorted, soc_inicial, charging_power_kw, battery_capacity_kwh):
    soc_actual = soc_inicial
    deficit_acumulado = 0
    
    for i, trip in enumerate(trips_sorted):
        # Base target from trip energy + buffer
        soc_objetivo_base = trip.energy_kwh / battery_capacity_kwh * 100 + safety_margin
        
        # Add propagated deficit from previous trip
        soc_objetivo = soc_objetivo_base + deficit_acumulado
        
        # Calculate if window allows reaching target
        if i < len(trips_sorted) - 1:
            next_trip = trips_sorted[i + 1]
            ventana_horas = calcular_ventana(trip.return_time, next_trip.departure_time)
            carga_posible = (charging_power_kw / battery_capacity_kwh * 100) * ventana_horas
            
            soc_al_siguiente_viaje = soc_objetivo + carga_posible
            soc_necesario_siguiente = next_trip.energy_kwh / battery_capacity_kwh * 100 + safety_margin
            
            if soc_al_siguiente_viaje < soc_necesario_siguiente:
                deficit_acumulado = soc_necesario_siguiente - soc_al_siguiente_viaje
            else:
                deficit_acumulado = 0
        
        # Calculate kWh needed for this trip
        trip.kwh_necesarios = (soc_objetivo - soc_actual) * battery_capacity_kwh / 100
        trip.soc_objetivo = soc_objetivo
        
        soc_actual = soc_objetivo  # For next iteration
    
    return trips
```

## 6. Sensor Update Triggers

### 6.1 Required Trigger Conditions

From the epic:

| Condition | Action |
|-----------|--------|
| SOC changes | If (home AND plugged) → recalculate profile |
| Home sensor → "on" | If plugged → recalculate profile |
| Plugged sensor → "on" | If home → recalculate profile |
| Trip added | Publish new trip to EMHASS |
| Trip modified | Update deferrable load parameters |
| Trip deleted | Remove from EMHASS, reassign indices |
| Home sensor → "off" | Stop monitoring (car not home) |
| Plugged sensor → "off" | Stop charging |

### 6.2 Implementation Approaches

**Option A: State Listeners in EMHASSAdapter**
```python
class EMHASSAdapter:
    def __init__(self, hass, ...):
        # Track state change listeners
        self._listeners = []
    
    async def async_setup_listeners(self):
        # Listen to SOC sensor
        self._listeners.append(
            async_track_state_change_event(
                self.hass, [self.soc_sensor_id], self._on_soc_changed
            )
        )
```

**Option B: TripManager Coordinates**
```python
class TripManager:
    async def async_setup(self):
        await self.vehicle_controller.async_setup()
        # After presence monitor ready, setup SOC listener
        if self._presence_monitor:
            await self._presence_monitor.async_setup_soc_listener(self)
```

**Option C: Dedicated Monitor (Recommended)**
```python
class SOCMonitor:
    def __init__(self, hass, trip_manager, emhass_adapter):

        self.emhass_adapter = emhass_adapter
    
    async def async_setup(self):

        ...

        # If yes, trigger profile republish
```

### 6.3 Debouncing Considerations

SOC sensors can be noisy (report frequently). Consider:
- Debounce: only trigger on stable change (e.g., 5% delta)
- Cooldown: not recalculate more than once per minute
- Batch: collect changes and recalculate once

## 8. Best Practices for EMHASS Integration

### 8.1 Index Management

1. **Assign indices at trip creation**, not at publish time
2. **Never reuse indices immediately** after deletion (EMHASS may still reference)
3. **Track index-to-trip mapping** in persistent storage
4. **Validate indices** before publishing (check trip still exists)

### 8.2 Power Profile Generation

1. **Generate 168 values** (7 days * 24 hours) for EMHASS
2. **Use timezone-aware datetimes** (HA's `dt_util.now()`)
3. **Set 0 for no charging**, positive for charging
4. **Group consecutive hours** for efficiency
5. **Include all active trips**, not just current day

### 8.3 Error Handling

1. **Store last error** with timestamp for diagnostics
2. **Update error status sensor** when failures occur
3. **Send notifications** for critical failures (EMHASS unavailable)
4. **Continue operation** if EMHASS fails (trips still managed locally)

### 8.4 Testing Considerations

1. **Mock EMHASS sensors** - don't need real EMHASS instance
2. **Test index stability** - delete/recreate trips, verify indices
3. **Test multi-trip scenarios** - verify window chaining
4. **Test SOC propagation** - verify deficit calculations
5. **Test trigger conditions** - SOC change, home/plugged changes

## 11. Open Questions Resolved

### Q1: Buffer Percentage
**A**: Use `safety_margin_percent` from `const.py` (default 10%) applied to battery capacity.

### Q2: Charging Rate for SOC
**A**: Calculate as `(charging_power_kw / battery_capacity_kwh) * 100` = % SOC per hour.

### Q3: Max Deficit Propagation
**A**: Limit to reasonable value (e.g., max 50% SOC above target) to prevent runaway. After limit, warn user.

### Q4: Car Leaves During Window
**A**: If `home_sensor` goes "off" during active window, pause tracking. When car returns, recalculate from new return time.

## 12. Summary: Key Patterns

1. **Index as stable identifier** - maintain across trip lifecycle
2. **Window from return to departure** - not from "now"
3. **SOC-based deficit propagation** - chain consecutive trips
4. **Trigger on (home AND plugged)** - only recalculate when conditions met
5. **Error isolation** - EMHASS failure doesn't stop trip management
6. **Persistent mapping** - index assignments survive restarts
7. **168-value profiles** - standard EMHASS planning horizon
