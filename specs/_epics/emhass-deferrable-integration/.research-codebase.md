# Research: Codebase Analysis

## Executive Summary

The EV Trip Planner is a Home Assistant custom component managing EV trips and optimizing charging schedules. The EMHASS integration publishes trips as deferrable loads with index-based assignment. The codebase follows HA 2026 patterns with async/await, strict typing, and runtime_data.

---

## 1. Module Boundaries

### 1.1 Core Modules

| File | Responsibility | Public API |
|------|----------------|------------|
| `__init__.py` | Integration entry, config flows, services registration, coordinator | `async_setup_entry()`, `register_services()`, `TripPlannerCoordinator` |
| `trip_manager.py` | Trip CRUD, energy calculations, power profile generation | `async_add_recurring_trip()`, `async_get_kwh_needed_today()`, `async_generate_power_profile()`, `async_generate_deferrables_schedule()` |
| `emhass_adapter.py` | EMHASS deferrable load publishing, index management | `async_publish_deferrable_load()`, `async_remove_deferrable_load()`, `publish_deferrable_loads()`, `get_assigned_index()` |
| `sensor.py` | HA sensor entities | `TripPlannerSensor`, `EmhassDeferrableLoadSensor`, `TripSensor` |
| `vehicle_controller.py` | Charging control strategies | `VehicleController`, `create_control_strategy()` |
| `presence_monitor.py` | Home/plugged status detection | `PresenceMonitor.async_check_charging_readiness()` |
| `schedule_monitor.py` | EMHASS schedule monitoring, charging control | `VehicleScheduleMonitor` |
| `const.py` | Configuration keys, defaults | `CONF_SOC_SENSOR`, `CONF_CHARGING_POWER`, `CONF_MAX_DEFERRABLE_LOADS` |
| `utils.py` | Helpers | `generate_trip_id()`, `calcular_energia_kwh()` |
| `dashboard.py` | Lovelace dashboard generation | `import_dashboard()` |
| `config_flow.py` | Config entry setup | ConfigEntriesFlowHandler |

### 1.2 Module Coupling Map

```
__init__.py
  ├── TripManager (trip_manager.py)
  │     ├── EMHASSAdapter (emhass_adapter.py) - via set_emhass_adapter()
  │     ├── VehicleController (vehicle_controller.py)
  │     └── uses: SOC sensor via hass.states.get()
  ├── TripPlannerCoordinator (uses TripManager for data updates)
  ├── EMHASSAdapter (standalone, initialized in async_setup_entry)
  └── Sensor entities (sensor.py)
        └── EmhassDeferrableLoadSensor -> TripManager for profile generation
```

### 1.3 Key Observations

1. **TripManager owns trip data but EMHASSAdapter owns index mapping**: The `EMHASSAdapter` maintains `self._index_map: Dict[str, int]` mapping trip_id to emhass_index, stored in `Store`. TripManager doesn't directly track which trips are published.

2. **EMHASSAdapter is set on TripManager but not stored within it**: `trip_manager.set_emhass_adapter(emhass_adapter)` wires them, but they remain separate objects.

3. **Sensor entities access TripManager directly**: `EmhassDeferrableLoadSensor` takes `trip_manager` in constructor and calls `async_generate_power_profile()` and `async_generate_deferrables_schedule()`.

4. **No SOC change listener exists**: The epic requires "when SOC changes while car is home and plugged, trigger recalculation" but there's no event listener for SOC changes in the current codebase.

## 3. Current Deferrable Load Patterns

### 3.1 TripManager Flow for Publishing

```python
# Trip added
async_add_recurring_trip() 
  -> async_create_trip_sensor()  # T034
  -> _async_publish_new_trip_to_emhass()  # T019.3
       -> emhass_adapter.async_publish_deferrable_load(trip)
       -> _publish_deferrable_loads()
            -> emhass_adapter.publish_deferrable_loads(all_trips, charging_power_kw)

# Trip updated
async_update_trip()
  -> async_update_trip_sensor()  # T035
  -> _async_sync_trip_to_emhass()  # T019.3
       -> emhass_adapter.async_update_deferrable_load(trip)
       -> _publish_deferrable_loads()

# Trip deleted
async_delete_trip()
  -> async_remove_trip_sensor()  # T034
  -> _async_remove_trip_from_emhass()
       -> emhass_adapter.async_remove_deferrable_load(trip_id)
       -> _publish_deferrable_loads()
```

### 3.2 EMHASSAdapter Index Management

- Pool of `max_deferrable_loads` (default 50) indices: `list(range(max_deferrable_loads))`
- Index assignment: `async_assign_index_to_trip(trip_id)` - assigns smallest available
- Index release: `async_release_trip_index(trip_id)` - returns index to pool
- Index persistence: via HA Store, survives restarts
- **No index reuse optimization**: If trip_1 (index 0) is deleted, trip_2 (index 1) keeps index 1 (doesn't get 0)

### 3.3 Sensor Entity Format

**EmhassDeferrableLoadSensor** (sensor.emhass_perfil_diferible_{entry_id}):
- State: "ready" | "error"
- Attributes:
  ```python
  {
      "power_profile_watts": List[float],  # 168 values (7 days * 24 hours)
      "deferrables_schedule": List[Dict],   # [{"date": "...", "p_deferrable0": "0.0", ...}, ...]
      "vehicle_id": str,
      "trips_count": int,
  }
  ```

**TripSensor** (sensor.trip_{trip_id}):
- State: "recurrente" | "pendiente" | "completado" | "cancelado"
- Attributes: trip_id, trip_type, descripcion, km, kwh, fecha_hora, activo, estado

### 3.4 Power Profile Calculation

In `trip_manager.py::async_generate_power_profile()`:
1. Loads all active trips (recurring where activo=True, punctual where estado=pendiente)
2. Sorts by deadline (earliest first)
3. For each trip, calculates energy needed from SOC and battery capacity
4. Sets charging power (watts) in the last hours before deadline

**Issue**: Current algorithm doesn't consider return time between trips - assumes charging window starts "now" rather than when car returns home.

### 3.5 Deferrables Schedule Format

In `trip_manager.py::async_generate_deferrables_schedule()`:
```python
{
    "date": "2026-03-29T14:00:00+01:00",
    "p_deferrable0": "0.0",  # or "3600.0"
    "p_deferrable1": "0.0",
    ...
}
```

## 4. Tech Stack Constraints

### 4.1 Home Assistant Version Requirements

- **Runtime data**: Uses `hass.data[DATA_RUNTIME]` pattern (compatible with HA 2024.4+)
- **Strict typing**: All files use `from __future__ import annotations` and type hints
- **Async/await**: All I/O operations are async
- **DataUpdateCoordinator**: Used for sensor data updates with 30-second fallback interval

### 4.2 Python Version

Assumed Python 3.10+ based on type alias syntax and dataclass usage.

### 4.3 Key HA APIs Used

- `homeassistant.core.HomeAssistant` - state access, services
- `homeassistant.helpers.storage.Store` - persistence
- `homeassistant.helpers.entity_registry` - sensor entity creation
- `homeassistant.config_entries.ConfigEntry` - config flow
- `homeassistant.helpers.event.async_track_state_change_event` - schedule monitoring
- `homeassistant.components.sensor.SensorEntity` - sensor base class

### 4.4 Presence Detection Sensors

The integration reads from user-configured sensors (not created by it):
- `home_sensor`: "on"/"off" - vehicle at home
- `plugged_sensor`: "on"/"off" - vehicle plugged in
- `charging_sensor`: "on"/"off"/"charging" - currently charging
- `soc_sensor`: percentage - battery SOC

### 4.5 EMHASS External Integration

EMHASS is a separate HA integration. The EV Trip Planner:
- Writes to template sensors (created by user via YAML)
- `sensor.emhass_deferrable_load_config_{index}` - per-trip config
- `sensor.emhass_perfil_diferible_{vehicle_id}` - aggregate profile
- Expects shell commands to POST to EMHASS API (user-configured)

## 5. Seam Identification

### 5.1 Natural Module Boundaries

| Seam | Description | Independence Level |
|------|-------------|-------------------|
| **TripManager ↔ EMHASSAdapter** | Already decoupled via adapter pattern, but TripManager calls EMHASS directly for publishing | Medium - can work independently |
| **EMHASSAdapter ↔ HA Sensors** | Adapter writes directly to `hass.states` | Tight coupling to HA state system |
| **Sensor Entities ↔ TripManager** | Sensors call TripManager methods for data | Tight - sensors need manager instance |
| **TripManager ↔ Presence** | Presence checks done within TripManager methods | Could be extracted |

### 5.2 Points for Independent Work

1. **SOC Change Listener**: New component that listens to SOC sensor state changes and calls `TripManager._publish_deferrable_loads()`. Could be added to `presence_monitor.py` or new file.

2. **Charging Window Calculator**: New module with `calcular_ventana_carga()` function. Inputs: trip data, SOC, return time. Output: window hours, sufficiency.

3. **SOC Milestone Algorithm**: New module with `calcular_hitos_soc()`. Inputs: sorted trips, initial SOC. Output: trips with updated kwh requirements.

4. **Index Stability Enhancement**: Modifications to `EMHASSAdapter` for stable index assignment when trips are deleted/reordered.

### 5.3 Shared State Creating Coupling

- **hass.data[DATA_RUNTIME]**: Shared namespace across all components
- **Store persistence**: Both TripManager and EMHASSAdapter use Store but with different keys
- **Config entry data**: Both read from `entry.data` for vehicle configuration

### 5.4 Data Flow for Trip Publishing

```
User adds trip via service call
    │
    ▼
TripManager.async_add_*_trip()
    │
    ├── Save to storage (async_save_trips)
    │
    ├── Create sensor entity (async_create_trip_sensor)
    │
    └── _async_publish_new_trip_to_emhass()
            │
            ├── EMHASSAdapter.async_publish_deferrable_load(trip)
            │       │
            │       ├── Assign index (or reuse)
            │       ├── Calculate params (kwh, hours, end_timestep)
            │       └── hass.states.async_set(config_sensor, "active", attrs)
            │
            └── EMHASSAdapter.publish_deferrable_loads(all_trips)
                    │
                    ├── _calculate_power_profile_from_trips()
                    └── hass.states.async_set(perfil_sensor, "ready", attrs)
```

## 7. Key Gaps Identified

1. **No SOC change listener**: Epic requires triggering recalc when SOC changes while home+plugged, but no listener exists.

2. **Charging window assumes "now"**: Current power profile calculation assumes charging can start immediately, not when car returns home.

3. **No deficit propagation**: No algorithm for propagating charge deficits between consecutive trips.

4. **Index reassignment on delete**: When a trip is deleted, indices are not reassigned to maintain EMHASS schedule stability.

5. **No return trip detection**: The system doesn't track when the car returns home to start the charging window.

6. **SOC used but not integrated with presence**: SOC is read but not coordinated with home/plugged status changes.
