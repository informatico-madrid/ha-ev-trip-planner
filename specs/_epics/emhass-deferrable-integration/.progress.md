# Epic: emhass-deferrable-integration

## Original Goal

Integración EMHASS Deferrable Load - organizar viajes en perfiles de carga aplazable con cálculo de ventanas de carga basado en SOC y generación de datos para EMHASS.

## Completed

- Created epic.md with 7 specs decomposition

## Learnings

### Key Decomposition Decisions

1. **SOC Integration as Baseline Spec**: Placed first because all subsequent calculations depend on having accurate SOC data. Trigger conditions: home + plugged + SOC change.

2. **Charging Window Calculation is Sequential**: Window = from car returns home until next trip start. Cannot calculate until we know when car returns (Spec 1) and when next trip is (Spec 2).

3. **SOC Milestone Algorithm Depends on Window Calculation**: The deficit propagation algorithm needs to know the charging window duration to calculate if the next trip can complete its charge. Example: 4h window × 10% SOC/h = 40% SOC gain.

4. **EMHASS Sensor Enhancement Depends on Milestone Algorithm**: The sensor needs to publish the corrected kWh requirements (with propagated deficits) to EMHASS.

5. **Trip Card and Automation Both Depend on EMHASS Sensor**: They read from the same enhanced sensor data.

6. **Integration Spec Last**: Gathers all previous specs together.

### Interface Contracts That Emerged

1. **`calcular_ventana_carga()`** - Returns window hours, kwh needed, sufficiency flag
2. **`calcular_hitos_soc()`** - Returns trips with updated kwh requirements and deficit propagation
3. **Sensor attributes** - `power_profile_watts`, `deferrables_schedule`, `trips_count`
4. **Automation triggers** - time_pattern at minutes 5 and 35, reads from `sensor.emhass_plan_{vehicle}_mpc_congelado`

### Risks Identified

1. **SOC sensor availability**: If SOC sensor is unavailable, calculations fall back to defaults. May need to handle gracefully.

2. **Index stability when trips are deleted**: Need to ensure index reassignment doesn't break EMHASS schedules. Current implementation reuses indices but doesn't guarantee stability.

3. **Cascade deficit propagation**: If there are many consecutive trips with deficits, the first trip's target SOC could grow very large. Need a sanity check/max limit.

4. **Window interruption**: If user leaves home during an active charging window and returns later, recalculation needs to handle partial windows.

### Dependencies Between Specs

```
Spec 1 (SOC Baseline + Return Time) → Spec 2 (Window Calc) → Spec 3 (Milestone) → Spec 4 (EMHASS Sensor)
                                                              ↓
                                              Spec 5 (Trip Card) ←→ Spec 6 (Automation)
                                                              ↓
                                              Spec 7 (Integration)
```

### Validation Findings Addressed

1. **Return-time detection added to Spec 1**: AC #4 expanded to record `hora_regreso` when car returns home. State stored: `hora_regreso` (datetime | None) and `soc_en_regreso`.

2. **Sensor naming mismatch clarified in Spec 6**: Two patterns exist - `sensor.emhass_plan_{vehicle}_mpc_congelado` (automation/epic) vs `sensor.emhass_deferrable{index}_schedule` (schedule_monitor.py). Epic automation uses the former, but implementation must support both or provide mapping.

3. **Battery capacity source clarified in Spec 3**: `battery_capacity_kwh` is REQUIRED parameter from `vehicle_config` (config entry data). Source documented in interface contracts.

4. **Index stability (soft delete) added to Spec 4**: AC #2 expanded with soft delete algorithm. Released indices go to `_released_indices` dict with timestamp, only reclaimed after cooldown period (default 24h).

5. **TripCard EMHASS access clarified in Spec 5**: TripSensor doesn't have EMHASS index directly. Must use `emhass_adapter.get_assigned_index(trip_id)` to resolve index for display.

### Key Files in Codebase

- `emhass_adapter.py`: Index management, publish methods, error handling
- `trip_manager.py`: Power profile generation, deferrables schedule, trip CRUD
- `sensor.py`: EmhassDeferrableLoadSensor, TripSensor
- `const.py`: CONF_CHARGING_POWER, DEFAULT_CHARGING_POWER, safety_margin_percent
- `presence_monitor.py`: May need SOC change listener integration

### EMHASS Format Requirements

```python
{
    "def_total_hours": [5.56],
    "P_deferrable_nom": [3600],
    "def_start_timestep": [0],
    "def_end_timestep": [18],
    "treat_deferrable_load_as_semi_cont": [true],
    "set_deferrable_load_single_constant": [true],
    "P_deferrable": [[0,0,3600,3600,...]]  # 168 values
}
```

### Milestone Algorithm Example (from user)

Sunday trips: morning 12:00, night 22:00
- Morning trip: needs 30% SOC + 10% buffer = 40% target
- Night trip: after 4h charging = 20% + 40% = 60% SOC, needs 80% → 20% deficit
- Deficit propagates to morning: morning target = 50% + 20% = **70%**
