# EMHASS Trip Charging Window Calculation Bug Fix Research

## Current Implementation (What's Broken)

### `async_publish_deferrable_load` - Single Trip Processing
The `async_publish_deferrable_load` method in `emhass_adapter.py` (lines 304-389) processes trips one at a time and has the following issues:

1. **Single Trip Processing**: Lines 363-369 create a single-trip charging window:
   ```python
   charging_windows = calculate_multi_trip_charging_windows(
       trips=[(deadline_dt, trip)],  # Only ONE trip!
       soc_actual=soc_current,
       hora_regreso=hora_regreso,
       charging_power_kw=self._charging_power_kw,
       duration_hours=6.0,
   )
   ```

2. **Missing Return Buffer**: The second sequential trip starts at timestep 0 instead of after the first trip's return buffer. This happens because:
   - Each call to `async_publish_deferrable_load` processes only one trip
   - The multi-trip function receives a list with only one trip
   - No consideration for subsequent trips' start times

3. **Inconsistent Processing**: The method has debug logging showing the problem.

## Multi-trip Function Analysis (How It Should Work)

### `calculate_multi_trip_charging_windows` Function
Located in `calculations.py` (lines 339-421), this function is designed for sequential trip processing:

1. **Function Signature**:
   ```python
   def calculate_multi_trip_charging_windows(
       trips: List[Tuple[datetime, Dict[str, Any]]],
       soc_actual: float,
       hora_regreso: Optional[datetime],
       charging_power_kw: float,
       duration_hours: float = 6.0,
   ) -> List[Dict[str, Any]]:
   ```

2. **Sequential Logic**:
   - **First trip** (idx=0): Starts at `hora_regreso` or `trip_departure_time - duration_hours`
   - **Subsequent trips** (idx>0): Start at `previous_arrival` (line 378)
   - **Chain reaction**: Each trip's window depends on the previous trip's completion

3. **Key Features**:
   - `previous_arrival` variable tracks when the previous trip ends (lines 365, 419)
   - Window start for trip N = arrival time of trip N-1
   - Properly handles charging duration and buffers

4. **Expected Behavior**:
   - Trip 1: 18:00-20:00 (if hora_regreso is 18:00)
   - Trip 2: 20:00-22:00 (starts when Trip 1 ends)
   - Trip 3: 22:00-00:00 (starts when Trip 2 ends)
