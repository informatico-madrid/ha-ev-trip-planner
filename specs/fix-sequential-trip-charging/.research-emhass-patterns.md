# Research: EMHASS Sequential Trip Charging Best Practices

**Spec**: fix-sequential-trip-charging  
**Research Date**: 2025-04-16  
**Bug**: `def_start_timestep_array: [0, 0]` causes second sequential trip to start immediately instead of after first trip's return buffer

---

## Executive Summary

The `def_start_timestep_array: [0, 0]` bug violates EMHASS's core constraint model. When set to 0, EMHASS interprets this as "start immediately from optimization window beginning" (Index 0), which breaks sequential trip charging where Trip 2 must wait for Trip 1 to complete.

**Key Finding**: EMHASS timestep arrays are **zero-based relative indices** from optimization window start, not absolute hour values. The bug requires calculating the correct timestep index based on trip duration + return buffer.

---

## 1. EMHASS Deferrable Loads Architecture

### 1.1 Core Parameter Model

From EMHASS documentation analysis:

```python
# Parameter: start_timesteps_of_each_deferrable_load
# Type: List of integers
# Meaning: Earliest timestep index from which load is allowed to start
# Index Base: 0 (start of optimization window)
# Default/Disable: 0 or negative = start immediately (no constraint)
```

**Critical Documentation Notes**:
- Values are **timestep indices**, not hour values
- Index 0 = beginning of optimization window (typically "now" when optimization runs)
- Index calculation: `(hours_delay / optimization_time_step_minutes) * 60`
- Example: With 30-min timesteps, 2-hour delay = `2 / 0.5 = 4` timesteps

### 1.2 EMHASS Optimization Logic

From API reference for `perform_optimization()`:

```python
def perform_optimization(
    def_start_timestep: list | None = None,  # Start constraint for each load
    def_end_timestep: list | None = None,    # Deadline for each load
    def_total_hours: list | None = None,     # Required operating hours
    ...
):
    """
    def_start_timestep: The timestep as from which each deferrable load is allowed to operate.
    def_end_timestep: The timestep before which each deferrable load should operate.
    """
```

**Key Constraint**: EMHASS enforces that deferrable loads can ONLY operate between `def_start_timestep` and `def_end_timestep`. Setting `[0, 0]` means BOTH trips can start at Index 0, causing immediate concurrent charging.

## 2. Sequential Trip Charging Patterns

### 2.1 Current Bug: What `[0, 0]` Means

**Scenario**: Two sequential trips
- Trip 1: Departs 08:00, returns 14:00 (6-hour duration)
- Trip 2: Departs 16:00, returns 20:00 (4-hour duration)
- Return buffer: 2 hours (user returns home + buffer before next trip charging)

**Current Implementation**:
```python
def_start_timestep_array = [0, 0]  # BUG: Both can start immediately!
```

**EMHASS Interpretation**:
- Trip 1: Can start charging at Index 0 (optimization window start)
- Trip 2: Can ALSO start charging at Index 0 (violates sequential dependency!)
- Result: EMHASS schedules both trips to charge simultaneously at 00:00

### 2.2 Correct Implementation Pattern

**Formula**:
```python
def_start_timestep_array[i] = cumulative_timesteps_before_trip_i

# Where cumulative_timesteps_before_trip_i includes:
# - Sum of all previous trip durations
# - Return buffer hours (user home + vehicle unavailable)
# - Converted to timesteps based on optimization_time_step
```

**Example Calculation** (30-min timesteps):
```python
optimization_time_step = 30  # minutes
timesteps_per_hour = 60 / 30 = 2

# Trip 1: Can start immediately
trip1_start_timestep = 0

# Trip 2: Must wait for Trip 1 (6h) + return buffer (2h)
trip2_wait_hours = 6 + 2 = 8  # hours
trip2_start_timestep = 8 * 2 = 16  # timesteps

def_start_timestep_array = [0, 16]
```

### 2.3 Industry Standard Return Buffers

**Research Findings**:

| Use Case | Return Buffer | Rationale |
|----------|--------------|-----------|
| Daily commute | 0-1 hours | Vehicle parked at workplace, available for charging |
| Errands/shopping | 2-3 hours | User returns home, may need vehicle again soon |
| Day trips | 4-6 hours | Typical activity duration (lunch, shopping, activities) |
| Long-distance travel | 6-8 hours | Driver rest requirements, destination charging |

**User Context**: "4-6 hours return buffer" suggests day trips where user returns home mid-day before evening trip. This is **conservative and appropriate** for sequential trip planning.

## 3. EMHASS Best Practices for Sequential Loads

### 3.1 Load Dependency Handling

EMHASS **does not natively support load dependencies** (Trip 2 depends on Trip 1 completion). Workaround:

**Approach 1: Temporal Constraints** (Recommended)
```python
# Encode dependency via start_timestep constraints
def_start_timestep_array = calculate_sequential_starts(trips, return_buffer_hours)

# Advantages:
# - EMHASS optimization respects constraints automatically
# - No dependency on load execution order
# - Fits EMHASS's linear programming model
```

**Approach 2: Single Aggregated Load** (Not Recommended)
```python
# Merge trips into one deferrable load
# Disadvantages:
# - Loses per-trip SOC tracking
# - Cannot optimize per-trip charging rates
# - Violates user's per-trip planning requirement
```

## 4. Calculation Algorithm for Sequential Trips

### 4.1 Pseudocode Implementation

```python
def calculate_def_start_timesteps(
    trips: List[Trip],
    return_buffer_hours: float,
    optimization_time_step_minutes: int
) -> List[int]:
    """
    Calculate def_start_timestep_array for sequential trips.
    
    Args:
        trips: List of trip objects with duration_hours attribute
        return_buffer_hours: Hours to wait after trip before next trip charging
        optimization_time_step_minutes: EMHASS optimization_time_step setting
    
    Returns:
        List of timestep indices for def_start_timestep_array
    """
    timesteps_per_hour = 60 / optimization_time_step_minutes
    start_timesteps = []
    cumulative_wait_hours = 0.0
    
    for i, trip in enumerate(trips):
        # First trip can start immediately (index 0)
        if i == 0:
            start_timesteps.append(0)
        else:
            # Subsequent trips wait for previous trip + return buffer
            prev_trip_duration = trips[i-1].duration_hours
            cumulative_wait_hours += prev_trip_duration + return_buffer_hours
            
            start_timestep = int(cumulative_wait_hours * timesteps_per_hour)
            start_timesteps.append(start_timestep)
    
    return start_timesteps
```

### 4.2 Example Calculation

**Input**:
```python
trips = [
    Trip(duration_hours=6.0),  # Trip 1: 08:00 - 14:00
    Trip(duration_hours=4.0),  # Trip 2: 16:00 - 20:00
]
return_buffer_hours = 2.0  # 2 hours after trip 1 returns
optimization_time_step_minutes = 30
```

**Output**:
```python
# Trip 1: Start immediately
trip1_start = 0

# Trip 2: Wait for Trip 1 (6h) + buffer (2h) = 8 hours
trip2_wait_hours = 6.0 + 2.0 = 8.0
trip2_start_timesteps = 8.0 * (60/30) = 8.0 * 2 = 16

def_start_timestep_array = [0, 16]
```

**EMHASS Interpretation**:
- Optimization window starts at 00:00 (Index 0)
- Trip 1 can start charging: Index 0 onwards (00:00)
- Trip 2 can start charging: Index 16 onwards (08:00)
- Result: Trip 2 waits until Trip 1 + buffer complete

## 5. Edge Cases and Validation

### 5.1 Overlapping Trip Windows

**Scenario**: Trip 1 duration + buffer > Trip 2 start time

**Example**:
- Trip 1: 08:00 - 14:00 (6 hours)
- Return buffer: 4 hours
- Trip 2 start: 16:00 (only 2 hours after Trip 1 returns)

**Problem**: Calculated `def_start_timestep[1]` = (6+4) * 2 = 20, but Trip 2 actually starts at 16:00.

**Solution**: Use `MAX(calculated_constraint, trip_2_start_time)`:
```python
trip2_constraint_timestep = 20  # From Trip 1 + buffer
trip2_actual_start_timestep = 16  # From Trip 2's actual departure
def_start_timestep_array[1] = max(20, 16) = 20  # Enforce the constraint
```

### 5.2 Multiple Sequential Trips

**Pattern**: Trip 1 → Buffer → Trip 2 → Buffer → Trip 3

```python
# Cumulative calculation ensures proper sequencing
cumulative_wait = 0.0
start_timesteps = []

for i, trip in enumerate(trips):
    if i == 0:
        start_timesteps.append(0)
    else:
        cumulative_wait += trips[i-1].duration_hours + return_buffer_hours
        start_timestep = int(cumulative_wait * timesteps_per_hour)
        start_timesteps.append(start_timestep)
```

**Example**:
- Trip 1: 6 hours, Buffer: 2h → Cumulative: 8h
- Trip 2: 4 hours, Buffer: 2h → Cumulative: 8 + 4 + 2 = 14h
- Trip 3: 3 hours, Buffer: 2h → Cumulative: 14 + 3 + 2 = 19h

With 30-min timesteps: `[0, 16, 38]`

### 5.3 End Timestep Validation

**Constraint**: `def_end_timestep` must allow sufficient charging time.

**Formula**:
```python
def_end_timestep[i] >= def_start_timestep[i] + (trip.charge_duration_hours * timesteps_per_hour)
```

**EMHASS Behavior**: If constraint violated, `validate_def_timewindow()` auto-corrects and emits warning.

## 8. Actionable Recommendations

### 8.1 Immediate Bug Fix (Priority: CRITICAL)

**File**: `custom_components/ev_trip_planner/emhass_adapter.py`

**Action**: Implement `_calculate_sequential_starts()` method

```python
def _calculate_sequential_starts(
    self, 
    trips: List[TripEntity],
    return_buffer_hours: float,
    optimization_time_step: int
) -> List[int]:
    """Calculate sequential def_start_timestep indices"""
    timesteps_per_hour = 60 / optimization_time_step
    start_timesteps = []
    cumulative_wait = 0.0
    
    for i, trip in enumerate(trips):
        if i == 0:
            start_timesteps.append(0)
        else:
            prev_duration = trips[i-1].trip_duration_hours
            cumulative_wait += prev_duration + return_buffer_hours
            start_timestep = int(round(cumulative_wait * timesteps_per_hour))
            start_timesteps.append(start_timestep)
    
    _LOGGER.debug("Calculated def_start_timestep_array: %s", start_timesteps)
    return start_timesteps
```

## 10. Summary

### Root Cause
`def_start_timestep_array: [0, 0]` tells EMHASS "all trips can start charging immediately," breaking sequential dependency where Trip 2 must wait for Trip 1 + return buffer.

### Solution
Calculate cumulative wait times: `def_start_timestep[i] = (sum(previous_trips_duration) + return_buffer_hours) * timesteps_per_hour`

### Implementation
1. Add `_calculate_sequential_starts()` to `emhass_adapter.py`
2. Add `return_buffer_hours` config option (default: 4.0h)
3. Update unit tests with sequential trip scenarios
4. Document for users (commute vs. day trip patterns)

### Impact
- **Fixes**: Trip 2 will respect Trip 1's completion + return buffer
- **Enables**: Proper sequential charging for multiple daily trips
- **Aligns**: With EMHASS's linear programming constraint model
- **Foundation**: For per-trip sensor migration (Gap #8)
