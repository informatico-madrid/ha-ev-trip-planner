### Open/Closed -- PARTIALLY COMPLIANT

- **Compliant**: `vehicle_controller.py` uses Strategy pattern with `VehicleControlStrategy` ABC (lines 83-99) and concrete implementations (SwitchStrategy, ServiceStrategy, ScriptStrategy, ExternalStrategy). New strategies can be added without modifying existing code.
- **Non-compliant**: Adding a new trip type requires modifying `TripManager.__init__` and all CRUD methods. The coordinator data contract (lines 34-43) is hardcoded. EMHASSAdapter has no extension points. Sensor definitions require modifying `definitions.py`.

### Liskov Substitution -- COMPLIANT (no issues found)

- No class inheritance hierarchy beyond standard HA entity inheritance.
- `FakeEMHASSPublisher` (`tests/__init__.py`, lines 74-99) implements `EMHASSPublisherProtocol` and passes as a valid substitute.
- All HA entity subclasses (`TripPlannerSensor`, `EmhassDeferrableLoadSensor`, `TripSensor`, `TripEmhassSensor`) properly override parent properties without behavioral surprises.

### Interface Segregation -- PARTIALLY COMPLIANT

- **Good**: Protocols in `protocols.py` are clean and focused:
  - `TripStorageProtocol`: 2 methods (`async_load`, `async_save`)
  - `EMHASSPublisherProtocol`: 4 methods (publish, remove, publish_all, update)
- **Bad**: `TripManager` exposes internal dicts as public attributes:
  - `self._recurring_trips` -- private-by-convention but accessed directly in tests and services (services.py:489, services.py:492, trip_manager.py:225)
  - `self._punctual_trips` -- same
  - `self.hass` -- full HA access from TripManager
  - `self._entry_id` -- stored directly, used for config entry lookups
- `EMHASSAdapter.__init__` accepts `entry` as `ConfigEntry | dict | anything` (lines 57-77), handling 3 different shapes via `isinstance`/`hasattr` checks. This is an interface that isn't segregrated -- it's one interface trying to serve multiple callers.

### Dependency Inversion -- NON-COMPLIANT

- **TripManager** depends on concrete `EMHASSAdapter` (not `EMHASSPublisherProtocol`). While the `set_emhass_adapter()` method accepts the protocol, the real adapter is a concrete `EMHASSAdapter` that implements the protocol via duck typing, not explicit Protocol adherence.
- **EMHASSAdapter** has direct dependency on `HomeAssistant` (`self.hass = hass`, line 59). It calls `hass.states.get`, `hass.states.async_remove`, `hass.states.async_set`, `hass.services.async_call`, `hass.config_entries.async_get_entry` -- all without abstraction.
- **sensor.py** imports and calls `hass.services.async_call` directly.
- **services.py** calls `hass.services.async_register`, `hass.services.async_call` directly.
- No inversion at the HA boundary -- the entire codebase is tightly coupled to `HomeAssistant` concrete type.

## 2. Architectural Anti-Patterns

### 2.1 God Objects
- **TripManager**: 2197 lines, 15+ responsibilities
- **EMHASSAdapter**: 2266 lines, 12+ responsibilities
- **services.py**: 1601 lines, 8+ responsibilities
- **sensor.py**: 962 lines, 6+ responsibilities

### 2.2 Global Mutable State
**File: `trip_manager.py`, line 45-46**
```python
_trip_manager_instance_count = 0
```
Global counter incremented in `__init__` (line 104). Creates state shared across all test instances.

### 2.3 Tight Coupling to Home Assistant
Every module directly imports and uses `HomeAssistant`:
- `trip_manager.py` line 21: `from homeassistant.core import HomeAssistant`
- `emhass_adapter.py` line 11: `from homeassistant.core import HomeAssistant`
- `sensor.py` line 22: `from homeassistant.core import HomeAssistant`
- Direct calls to `hass.states.get`, `hass.states.async_remove`, `hass.states.async_set`, `hass.services.async_call` with no abstraction layer.

### 2.4 Circular Dependencies
The data flow forms a circle:
```
TripManager -> EMHASSAdapter (publish trips)
EMHASSAdapter -> Coordinator (update coordinator.data)
Coordinator -> TripManager (read trips in _async_update_data)
EMHASSAdapter -> TripManager (inject _presence_monitor via coordinator)
```

### 2.5 Implicit Dependencies
**EMHASSAdapter._presence_monitor** (`emhass_adapter.py`, lines 815-821, 1144-1149):
```python
if self._presence_monitor is None and coordinator is not None:
    trip_manager = getattr(coordinator, "_trip_manager", None)
    if trip_manager and hasattr(trip_manager, "vehicle_controller"):
        vc = trip_manager.vehicle_controller
        if vc and hasattr(vc, "_presence_monitor"):
            self._presence_monitor = vc._presence_monitor
```
This code traverses 4 levels of object graph to discover `_presence_monitor`. It runs inside `async_publish_all_deferrable_loads` and `publish_deferrable_loads`. If `coordinator` is None or `_trip_manager` doesn't exist, the presence monitor is silently unavailable, and `hora_regreso` becomes `None`, changing the charging window calculation behavior.

### 2.6 Race Condition Sources

**Race 1: Coordinator refresh vs manual data mutation**
- Coordinator has 30-second update interval (`coordinator.py`, line 67)
- `TripManager.publish_deferrable_loads` calls `coordinator.async_refresh()` (`trip_manager.py`, line 269)
- `EMHASSAdapter.async_publish_all_deferrable_loads` directly sets `coordinator.data` (`emhass_adapter.py`, lines 737-743)
- `EMHASSAdapter.async_cleanup_vehicle_indices` directly sets `coordinator.data` (`emhass_adapter.py`, lines 1851-1861)
- **Result**: Two concurrent writers to `coordinator.data` with no synchronization. `_async_update_data` (the coordinator's update method) may read trips while EMHASSAdapter is simultaneously clearing `per_trip_emhass_params`.

**Race 2: Delete trip -- cache clearing vs coordinator refresh**
Sequence during `async_delete_trip` (trip_manager.py, lines 747-780):
1. Delete trip from `_recurring_trips` / `_punctual_trips`
2. Call `async_save_trips()`
3. Call `async_remove_trip_sensor()`
4. Call `_async_remove_trip_from_emhass()`
   a. Calls `_emhass_adapter.async_remove_deferrable_load(trip_id)`
   b. Calls `self.publish_deferrable_loads()` (no args)
      - This calls `_get_all_active_trips()` which reads from memory
      - Then calls `_emhass_adapter.async_publish_all_deferrable_loads(trips)`
      - This iterates remaining trips, populates per-trip cache
5. Refresh coordinator

The bug: In step 4b, `_get_all_active_trips()` reads remaining trips from memory. If the coordinator's `_async_update_data` runs concurrently (30s interval), it also reads from memory. The coordinator's data may show the trip list AFTER deletion, but the EMHASS cache may still contain stale entries from a previous publish. The coordinator then reads the stale `per_trip_emhass_params` from EMHASSAdapter's cache (which hasn't been cleared for the deleted trip).

**Race 3: Cascade deletion -- multiple caches cleared at different times**
In `async_delete_all_trips` (trip_manager.py, lines 782-853):
1. Clear trip dicts
2. Call `async_save_trips()`
3. Clear `emhass_adapter._published_trips`, `_cached_per_trip_params`, `_cached_power_profile`, `_cached_deferrables_schedule`
4. Call `publish_deferrable_loads([])`
5. Directly clear `emhass_adapter` caches AGAIN (extra safeguard)
6. Directly set `coordinator.data` keys to empty

There are 6 different clearing operations across 3 objects (`trip_manager`, `emhass_adapter`, `coordinator`). Some are async (awaited), some are direct dict mutations. The async ones complete asynchronously, so the coordinator refresh may see a partially-cleared state.

### 2.7 Ambiguous Data Flows

**Where does data come from?**
- Trip data: `TripManager._recurring_trips` / `_punctual_trips` (in-memory) -> `YamlTripStorage` (persistent) -> loaded on startup via `_load_trips()`
- EMHASS params: Computed by `EMHASSAdapter._populate_per_trip_cache_entry()` -> stored in `EMHASSAdapter._cached_per_trip_params` -> exposed via `get_cached_optimization_results()` -> read by `TripPlannerCoordinator._async_update_data()` -> consumed by `EmhassDeferrableLoadSensor.extra_state_attributes`
- SOC: Read from `hass.states.get(soc_sensor)` in `EMHASSAdapter._get_current_soc()` -- depends on external sensor state that tests cannot control

**Where does data go?**
- Coordinator data: Mutated by `_async_update_data` (on 30s interval) AND directly by `EMHASSAdapter` methods. No single source of truth.
- EMHASS sensor attributes: Built from `coordinator.data["per_trip_emhass_params"]`, which is populated by EMHASSAdapter cache, which is updated by batch calculation in `async_publish_all_deferrable_loads`.
- Sensor states: Updated by coordinator `async_refresh`, which triggers `_async_update_data` which calls `async_publish_deferrable_loads()` on `TripManager` which calls `_emhass_adapter.async_publish_all_deferrable_loads()`. This is a loop.

### 2.8 Duplicate Cache Clearing (Defensive Coding Anti-Pattern)
The code has excessive "extra safeguard" clearing:
- `trip_manager.py` lines 803-808: Clear adapter caches BEFORE `publish_deferrable_loads`
- `trip_manager.py` lines 816-824: Clear adapter caches AGAIN after `publish_deferrable_loads`
- `emhass_adapter.py` lines 722-728: Clear caches in `async_publish_all_deferrable_loads` when trips is empty
- `emhass_adapter.py` lines 1053-1061: Clear caches in `publish_deferrable_loads` when trips is empty
- `emhass_adapter.py` lines 1827-1835: Clear caches in `async_cleanup_vehicle_indices`
- `emhass_adapter.py` lines 1947-1979: Reset coordinator.data keys AGAIN after force-setting state

This pattern indicates the developers are aware of race conditions and are throwing cache clears at the problem rather than fixing the root cause.

## 3. Testability Score: 3/10

### Justification

**Positive factors:**
- Protocols exist for `TripStorageProtocol` and `EMHASSPublisherProtocol`
- `FakeTripStorage` and `FakeEMHASSPublisher` exist in tests
- `create_mock_trip_manager` and `create_mock_coordinator` factory functions exist
- Strategy pattern in `vehicle_controller.py` is testable

**Negative factors:**
1. **Home Assistant coupling** (severity: CRITICAL): Every class directly depends on `HomeAssistant`. The E2E tests must run a real HA instance because there is no way to mock `hass.states.get`, `hass.services.async_call`, `hass.config_entries.async_entries`, `hass.config_entries.async_get_entry`, `er.async_get`, etc.

2. **Async timing uncontrollability** (severity: CRITICAL): The coordinator's 30-second refresh interval (`coordinator.py`, line 67) competes with manual `async_refresh()` calls (`trip_manager.py`, line 269) and direct data mutations (`emhass_adapter.py`, line 737). Tests cannot predict which writer will win.

3. **SOC dependency** (severity: HIGH): Charging decisions depend on `EMHASSAdapter._get_current_soc()` which reads from an external sensor. Tests configure SOC=80 in `TEST_CONFIG` (`tests/__init__.py`, line 17), but the real HA instance may have a different SOC sensor value.

4. **Entity ID non-determinism** (severity: HIGH): Entity IDs are constructed from `coordinator.vehicle_id` which is derived from `entry.data["vehicle_name"]` via normalization (`coordinator.py`, lines 72-76). The normalization is `vehicle_name.lower().replace(" ", "_")`. If onboarding sends "Test Vehicle" vs "test_vehicle", the entity ID changes.

5. **Multiple concurrent data paths** (severity: HIGH): Service call -> TripManager -> EMHASSAdapter -> coordinator.data -> sensor.attributes. Any delay in any step causes the test to see stale data. The `waitForTimeout()` calls in tests are workarounds for this structural problem.

6. **`hass.async_block_till_done()` dependency** (severity: MEDIUM): Tests must call this after every service call to flush async operations. Without it, sensor data is stale. This is not an explicit async boundary -- it's a race condition workaround.

7. **Debug log pollution** (severity: MEDIUM): 150+ `LOGGER.warning` calls and 50+ "E2E-DEBUG" log statements make it impossible to correlate test failures with actual code behavior.

8. **Sensor async_add_entities callback** (severity: MEDIUM): Dynamic sensor creation uses `async_add_entities` callback stored in `runtime_data.sensor_async_add_entities` (`__init__.py`, line 177). This callback's timing is controlled by HA's entity platform, not by the code that calls it.

## 4. Root Cause Analysis of E2E Test Fragility

### 4.1 Why did the coordinator race condition happen?

**Design flaw, not a bug.** The coordinator was designed with a 30-second periodic refresh (`coordinator.py`, line 67), but service handlers immediately trigger `coordinator.async_refresh_trips()` after every CRUD operation (`services.py`, lines 50, 67, 129, 212, 225, 246, 259, 272, 285, 298). This means:

1. Service call modifies trip state
2. Service calls `coordinator.async_refresh_trips()` -> `coordinator.async_refresh()`
3. Coordinator starts `_async_update_data()` which calls `_emhass_adapter.get_cached_optimization_results()`
4. Meanwhile, EMHASS adapter may have been updating its cache in a different path
5. The coordinator reads stale cache data

The fix should have been to have `async_refresh` wait for the current update cycle to complete, or to have a single writer for `coordinator.data`. Instead, there are two writers: `_async_update_data` and direct `coordinator.data` mutations in EMHASSAdapter.

### 4.2 Why is `waitForTimeout()` needed in tests?

Because there is **no synchronous async boundary**. The chain of async operations is:

```
service call -> TripManager CRUD -> async_save_trips() -> 
EMHASS adapter publish -> cache update -> coordinator.async_refresh() -> 
_async_update_data -> sensor attributes re-read
```

None of these operations have a single completion point that a test can await. The `hass.async_block_till_done()` in HA is not sufficient because the coordinator's `_async_update_data` may not have completed when the test reads sensor state.

The missing boundary would be: a single async method that the service handler awaits, which completes only when all side effects (storage, EMHASS publish, coordinator update, sensor rebuild) are confirmed done.

### 4.3 Why were hardcoded entity IDs needed?

Because entity naming depends on `coordinator.vehicle_id`, which is derived from `entry.data["vehicle_name"]` via `normalize_vehicle_id()`. During E2E test onboarding, the vehicle name is sent via the UI. If the UI sends "Test Vehicle" and the normalization produces "test_vehicle", the entity ID would be `sensor.ev_trip_planner_test_vehicle_kwh_today`. But if the onboarding sends a different name format, the entity ID changes. Tests cannot rely on the onboarding flow to produce a deterministic name, so they hardcode the entity ID pattern.

The normalization function (`utils.py`, referenced from services.py:21) converts to lowercase and replaces spaces with underscores. But the onboarding flow may send the vehicle name in a different format than expected (e.g., "Test Vehicle" vs "test_vehicle" vs "Test_Vehicle"), producing different entity IDs.

### 4.4 Why was SOC level critical for test results?

Because the EMHASS adapter's `determine_charging_need` function (`emhass_adapter.py`, lines 552-555) uses SOC to decide whether a trip needs charging:

```python
decision = determine_charging_need(
    trip, soc_current, battery_capacity_kwh,
    charging_power_kw, safety_margin_percent,
)
```

If `soc_current` >= `soc_objetivo` for all trips, then `decision.needs_charging` is False, and `power_profile` becomes `[0.0] * 168` (line 629). This means:
- `def_total_hours_array` becomes `[0]` or is missing
- `p_deferrable_matrix` becomes `[]`
- The EMHASS sensor shows empty charging schedule

When the test expects a non-empty `def_total_hours_array` but SOC is already high, the assertion fails. The SOC is read from the real sensor in the HA instance (`emhass_adapter.py`, lines 2236-2237), which the test cannot control.

## 5. Recommendations

### Priority 1: Fix the coordinator data flow (prevents ALL other issues)

**Problem**: `coordinator.data` is written by two independent paths: `_async_update_data()` (every 30s) and direct mutation by EMHASSAdapter.

**Fix**: Make the coordinator the SINGLE writer of `coordinator.data`. Remove all direct mutations (`emhass_adapter.py` lines 737-743, 1847-1861, 1951-1956). Instead, have EMHASSAdapter publish to an internal event queue or callback that the coordinator processes in `_async_update_data`.

**Impact**: Eliminates the race condition between periodic refresh and manual refresh. Eliminates the need for `waitForTimeout()` to wait for coordinator refresh.

### Priority 2: Break up TripManager (prevents God object issues)

**Problem**: TripManager has 15+ responsibilities, making it impossible to test in isolation.

**Fix**: Extract into:
- `TripStore`: CRUD + storage (injectable `TripStorageProtocol`)
- `TripScheduler`: recurring trip rotation, next-trip calculation
- `TripService`: orchestrates TripStore + TripScheduler + TripValidator
- `ChargingCalculator`: SOC calculation, charging window calculation, deficit propagation (pure functions, already partially in `calculations.py`)
- `PowerProfileGenerator`: power profile + deferrable schedule generation (delegates to `calculations.py`)

Each extracted class would have 2-3 responsibilities and be independently testable.

### Priority 3: Abstract Home Assistant at the module boundary

**Problem**: Every module directly calls HA APIs, making unit testing impossible.

**Fix**: Create a `HomeAssistantGateway` abstraction that wraps all HA calls (`states.get`, `services.async_call`, `config_entries.async_get_entry`, etc.). Inject this into TripManager, EMHASSAdapter, and sensor modules. The real implementation uses HA APIs; the test implementation returns controlled values.

### Priority 4: Create a single async completion boundary for service handlers

**Problem**: Service handlers call async methods but don't have a single await point for all side effects.

**Fix**: Each service handler should call a single orchestrator method that awaits:
```python
async def handle_delete_trip(call):
    result = await trip_service.delete_trip(vehicle_id, trip_id)
    await result.complete  # All side effects confirmed
```

### Priority 5: Make SOC configurable for tests

**Problem**: `EMHASSAdapter._get_current_soc()` reads from a real HA sensor.

**Fix**: Add a `soc_override` parameter to `EMHASSAdapter` constructor. When set, use the override instead of reading from HA sensor. This allows tests to control SOC deterministically.

### Priority 6: Standardize entity naming

**Problem**: Entity IDs depend on normalized vehicle name from config entry, which varies based on how the user enters it.

**Fix**: Use `entry.entry_id` (always unique, never changes) as the primary key for entity IDs, not `vehicle_id` derived from `vehicle_name`. The vehicle name can be used for display only.

## Key Finding

**The E2E test fragility is a DIRECT RESULT of architectural violations, not test quality.** The tests are working correctly against a system that was not designed for testability. The race conditions, timing dependencies, and non-deterministic entity IDs are all symptoms of:
1. God objects with too many responsibilities
2. Missing abstraction at the HA boundary
3. Multiple concurrent writers to shared state
4. No synchronous async completion boundaries

Test reliability cannot be achieved without architectural fixes.
