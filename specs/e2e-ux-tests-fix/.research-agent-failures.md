# E2E Test AI Agent Failure Analysis — Root Cause Report

**Date:** 2026-04-22
**Project:** EV Trip Planner (Home Assistant Custom Component)
**Model:** Qwen3.6-35B-A3B-FP8 (200K context window, system-capped at 131K)

---

## Executive Summary

Multiple AI agent loops (Ralph) failed to produce working E2E tests for the EV Trip Planner Home Assistant integration. The failures fell into three independent categories:

1. **Framework crash**: Ralph loops accumulated context until the model's 131K token limit was exceeded, causing HTTP 500 errors.
2. **Inherently brittle test approach**: Tests relied on `page.evaluate()` to read `hass.states` from the browser — an implementation detail of HA's frontend JavaScript that is non-deterministic and fragile.
3. **Async timing race conditions**: The data flow from trip creation through UI update spans multiple async boundaries (TripManager → EMHASS cache → Coordinator → HA state machine → WebSocket → browser `hass.states`). Tests used `waitForTimeout()` as a workaround for fundamental non-determinism.

**Key insight**: The AI agents wrote functionally correct test code against a system with inherently non-deterministic async timing. The tests were "broken" because they could not guarantee the browser's `hass.states` would be updated within the expected time window, regardless of how long they waited.

## Categorized Failure Modes

### 1. Framework/Crash Failures

**What happened**: Ralph loops exceeded the model's 131K token context limit, causing the Qwen API to return HTTP 500.

**Root cause**: The Ralph framework accumulates conversation context across loops. After ~15+ turns, input tokens reached ~99K. With the model's max output of 32K, total required tokens would be ~131K+, exceeding the 131,072 limit.

**Why it matters**: This is a Ralph framework configuration issue, not a test code issue. The model has a 200K window but the system was capped at 131K. As Ralph loops accumulate context, the prompt grows until it crashes.

**Evidence**:
- `claude_output_2026-04-22_15-00-33.log` — Successful loop, 1.29M total input tokens across turns, but within per-turn limit
- `claude_output_2026-04-22_15-38-16.log` — Failed loop, 99K input tokens in a single prompt, API error 500

**Fix**: Configure Ralph to use the full 200K window, implement context pruning between loops, or set lower output token limits.

### 2. Test Code Failures

#### 2.1. Browser `hass.states` Access via `page.evaluate()`

**File**: `tests/e2e/emhass-sensor-updates.spec.ts` (667 lines)

**Problem**: Tests read `hass.states` from the browser's HA JavaScript object via `page.evaluate()`. This is an implementation detail of Home Assistant's frontend SPA.

```typescript
// Example pattern from tests:
const state = await page.evaluate(() => {
  const hass = (window as any).hass;
  return hass.states['sensor.xxx'];
});
```

**Why it fails**:
1. HA's frontend `hass.states` is updated via WebSocket with non-deterministic timing
2. The state object may not exist yet (entity not registered) or may have stale data
3. `page.evaluate()` executes in the browser context — it can only see what the JS SPA has received
4. There is no synchronization point between the backend (sensor updated) and frontend (browser received update)

**Evidence in test code**: Multiple `page.waitForTimeout(3000)` and `page.waitForTimeout(5000)` calls as workarounds for this fundamental timing gap.

#### 2.2. Hardcoded Entity IDs

**Problem**: Tests use fragile hardcoded entity IDs like `'sensor.ev_trip_planner_test_vehicle_emhass_perfil_diferible_test_vehicle'`. These are generated dynamically by the integration and depend on vehicle naming conventions, EMHASS sensor type, and configuration.

**Impact**: If the entity naming convention changes (even slightly), all tests break. The sensor ID format is:
```
sensor.{vehicle_name}_emhass_perfil_diferible_{vehicle_name}
```

#### 2.3. Navigating to HA Internal Pages

**Problem**: Tests navigate to `/developer-tools/state` and `/config/devices` to assert on sensor data. These HA pages:
- Use shadow DOM (`ha-panel-devtools`, `ha-panel-config`)
- Render data via `ha-data-table` (web component, not plain HTML)
- Use client-side JavaScript to populate tables from `hass.states`
- Dynamic content is not guaranteed to be rendered within any specific timeout

**Evidence**: `getByRole('textbox', { name: /filter/i })` — relies on HA's filter input having a specific accessible name, which may not be consistent across HA versions.

#### 2.4. Text Assertions on HA-Rendered Content

**Problem**: Assertions check for text like `'power_profile_watts:'` in the DOM. This depends on:
- How HA renders sensor attributes on its pages
- Whether the attribute name uses underscores or camelCase (HA frontend may transform)
- The exact HTML structure of `ha-data-table` rows

### 3. Implementation/Async Timing Failures

#### 3.1. The Full Async Data Flow

The data flow from trip creation to UI update spans these async steps:

```
User clicks "Create Trip" in UI
  → HA service call (services.py)
  → TripManager.async_add_punctual_trip() or async_add_recurring_trip()
  → TripManager saves to storage (JSON file)
  → TripManager creates sensor entity (registry callback)
  → EMHASS adapter.async_publish_all_deferrable_loads()
    → EMHASS adapter populates _cached_per_trip_params
    → EMHASS adapter populates _cached_power_profile
  → TripManager.publish_deferrable_loads()
    → coordinator.async_refresh() (immediate) or async_request_refresh() (debounced)
  → TripPlannerCoordinator._async_update_data()
    → Reads from TripManager
    → Reads from EMHASS adapter.get_cached_optimization_results()
  → Coordinator publishes updated data to all listeners
  → Sensor entity async_write_ha_state()
  → HA state machine updates hass.states
  → WebSocket broadcast to all connected frontend clients
  → Browser HA JS object (window.hass) receives update
  → window.hass.states is updated
  → page.evaluate() can read the new state
```

Each step is async and non-deterministic. Steps 6-7 (coordinator refresh) are the most variable:
- The coordinator uses a 30-second polling interval as baseline
- `async_refresh()` is immediate but the sensor update pipeline (steps 8-12) still takes time
- `async_request_refresh()` is debounced (min 1 second), causing additional delay

**Evidence in integration code**:
- `coordinator.py`: 30-second interval, `async_refresh()` for immediate updates
- `emhass_adapter.py`: Multiple "directly update coordinator.data" hacks to work around timing
- `trip_manager.py`: `async_delete_all_trips()` directly sets `coordinator.data` as a workaround
- `emhass_adapter.py`: `async_cleanup_vehicle_indices()` directly sets `coordinator.data`

#### 3.2. Coordinator Refresh Debouncing

**File**: `custom_components/ev_trip_planner/coordinator.py` (180 lines)

The `DataUpdateCoordinator` has both `async_refresh()` (immediate) and `async_request_refresh()` (debounced). The integration code uses both, creating unpredictable timing:

- Some code paths call `async_refresh()` directly
- Others use `async_request_refresh()` which is debounced
- The test cannot know which path was taken, making `waitForTimeout()` unreliable

#### 3.3. EMHASS Cache Synchronization

**File**: `custom_components/ev_trip_planner/emhass_adapter.py` (2266 lines)

The EMHASS adapter caches optimization results in `_cached_per_trip_params`, `_cached_power_profile`, `_cached_deferrables_schedule`, and `_cached_emhass_status`. These are populated by `async_publish_all_deferrable_loads()` and read by the coordinator via `get_cached_optimization_results()`.

**Critical timing gap**: If the coordinator's `_async_update_data()` runs before `async_publish_all_deferrable_loads()` completes, the sensor will show stale/empty data. There is no explicit synchronization between these two code paths.

**Evidence of "fixes"**: The integration code contains multiple ad-hoc "directly update coordinator.data" patterns to work around this timing issue — a clear sign that the normal async flow is unreliable.

- `input_number.test_vehicle_soc` — SOC backing for tests

If any of these don't match what the config flow expects, setup fails silently.

#### 4.3. Panel Loading Race

**File**: `tests/e2e/trips-helpers.ts`

`navigateToPanel()` uses retry logic:
```typescript

```

The panel JS (`panel.js`) loads asynchronously from HA's frontend. The retry logic (3 attempts total) is itself fragile — if the panel takes longer than expected to load (slow server, large panel bundle), the test fails before any real assertions run.

## Root Cause Analysis

### The Fundamental Problem

The AI agents wrote tests against the **wrong abstraction layer**. They tested the Home Assistant frontend's `hass.states` JavaScript object, which is:
1. An implementation detail of HA's frontend (not a public API)
2. Updated asynchronously via WebSocket (non-deterministic timing)
3. Only updated after the full backend → frontend pipeline completes
4. Subject to HA version changes (frontend behavior may change)

### What Would Be Reliable

**Option A: Test via HA REST API (backend-focused)**
- Use `page.goto('/api/ev-trip-planner/...')` or direct `fetch()` to test the backend
- Assert on backend response data, not on frontend rendering
- Bypasses all frontend timing issues

**Option B: Test via HA service calls and wait for state via API**
- Use `page.evaluate()` to call HA's service API (`/api/services/{domain}/{service}`)
- Poll `/api/states/{entity_id}` directly (REST API, not WebSocket)
- REST API returns definitive state — no WebSocket timing uncertainty

**Option C: Integration tests without browser**
- Use pytest to test the Python integration directly
- No browser, no WebSocket, no frontend
- Fast, reliable, deterministic
- This is what `create-trip.spec.ts` does for the simpler parts

### Why the AI Agents Missed This

1. **The example tests (`create-trip.spec.ts`) used the same `page.evaluate()` pattern** — the AI had a reference implementation that appeared to work, so it assumed the pattern was valid
2. **`create-trip.spec.ts` only tests the UI panel** — it asserts on visible text elements in the panel, which are rendered synchronously after user interaction. The sensor tests go deeper into HA internals
3. **The async timing gap is invisible in code** — `waitForTimeout(3000)` looks like it should work. The agent has no way to know the backend → frontend pipeline might take 5 seconds on a slow system
4. **Context accumulation** — as Ralph loops accumulated more context, the model had less room for nuanced analysis and defaulted to the patterns it already knew

## Recommended Fix Strategy

### Short Term: Fix the Test Approach

1. **Replace `page.evaluate(() => hass.states[...])` with REST API polling**
   - Use `page.waitForResponse()` or explicit polling of `/api/states/{entity_id}`
   - This is a deterministic API — no WebSocket timing uncertainty

2. **Replace HA page navigation assertions with API assertions**
   - Instead of navigating to `/developer-tools/state` and checking DOM text
   - Use `fetch('/api/states/sensor.xxx')` from `page.evaluate()` to get definitive state
   - Assert on the JSON response data structure

3. **Use entity_id lookups instead of hardcoded IDs**
   - Discover entity IDs via the HA REST API (`/api/states` filter by `entity_id` pattern)
   - Or use the HA config API (`/api/config/config_entries`) to find integration entities

### Medium Term: Fix the Integration Code

1. **Unify coordinator refresh strategy**
   - Always use `async_refresh()` after trip CRUD (not `async_request_refresh()`)
   - Remove ad-hoc "directly update coordinator.data" hacks

2. **Add explicit synchronization point**
   - After `async_publish_all_deferrable_loads()`, add a callback or event that signals "cache ready"
   - Coordinator should wait for this before publishing to listeners

3. **Add entity registry lookup helper**
   - Provide a deterministic way to find sensor entity IDs by vehicle name
   - This makes tests resilient to entity ID naming changes

### Long Term: Split Test Layers

1. **Unit tests** (pytest) — test Python logic, no browser needed. Already passing at 1636/1636
2. **API tests** (pytest + aiohttp) — test HA service calls and responses, no UI
3. **UI tests** (Playwright) — test only panel rendering and user interaction, not sensor data

This separation would prevent the current hybrid approach that tries to assert on sensor state through the browser, which is the root of the fragility.
